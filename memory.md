# โปรแกรมทำเอกสารโครงการก่อสร้าง — ไฟล์ความจำโปรเจกต์

อัปเดตล่าสุด: 2026-07-17

## เป้าหมายโปรเจกต์
สร้างโปรแกรม "กรอกข้อมูล → ออกเอกสารราชการ .docx" สำหรับงานเอกสารโครงการก่อสร้าง
ตามระเบียบสำนักนายกรัฐมนตรีว่าด้วยงานสารบรรณ พ.ศ. 2526 (และฉบับแก้ไขถึง พ.ศ. 2564)
รูปแบบ: เว็บฟอร์ม (HTML เดี่ยว รันในเบราว์เซอร์ ไม่ต้องมีเซิร์ฟเวอร์) → ดาวน์โหลดไฟล์ Word (.docx)
ฟอนต์มาตรฐาน: TH Sarabun New ต้องมีหัวกระดาษตราครุฑตามระเบียบ

ผู้ใช้จะทยอยหาแบบฟอร์มตัวอย่าง (เอกสารราชการ/เอกสารโครงการก่อสร้าง) มาเพิ่มทีหลัง —
โครงสร้างโฟลเดอร์จึงถูกออกแบบให้ขยายเพิ่มประเภทเอกสารได้เรื่อยๆ โดยไม่ต้องรื้อของเดิม

## โครงสร้างโฟลเดอร์ (ในโฟลเดอร์ที่ผู้ใช้เลือก ConsDocx/)

```
ConsDocx/
├── memory.md                  ← ไฟล์นี้ (บริบท/การตัดสินใจของโปรเจกต์)
├── shared/
│   └── docx.lib.js            ← docx npm package (browser IIFE build) — ใช้ร่วมกันทุกโมดูล
└── บันทึกข้อความ/
    └── index.html             ← โมดูลแรก: ฟอร์ม + ตัวสร้างไฟล์ .docx สำหรับบันทึกข้อความ
```

**กติกาการขยาย:** เอกสารแต่ละประเภท (เช่น หนังสือภายนอก, ประกาศ, คำสั่ง, สัญญาก่อสร้าง,
แบบขออนุมัติเบิกจ่าย ฯลฯ) ให้สร้างเป็นโฟลเดอร์ใหม่ระดับเดียวกับ `บันทึกข้อความ/`
แต่ละโฟลเดอร์มี `index.html` ของตัวเอง และ `<script src="../shared/docx.lib.js">`
อ้างอิงไลบรารีชุดเดียวกัน จะได้ไม่ต้องก็อปไฟล์ 1MB+ ซ้ำทุกโฟลเดอร์

## การตัดสินใจทางเทคนิค

- **ไลบรารีสร้าง .docx:** ใช้ npm package `docx` (v9.7.1) — build แบบ `dist/index.iife.js`
  เพื่อรันในเบราว์เซอร์ล้วนๆ ผ่าน `<script src>` ธรรมดา ไม่ต้องมี build step / bundler
  global variable ที่ได้คือ `docx` (เช่น `docx.Document`, `docx.Packer.toBlob(...)`)
- **เหตุผลที่เลือกไฟล์ local แทน CDN:** แซนด์บ็อกซ์นี้ไม่มีสิทธิ์ทดสอบ fetch จาก CDN ได้ชัดเจน
  และการฝังไฟล์ไว้ในเครื่องผู้ใช้ทำให้โปรแกรม **ใช้งานได้แบบออฟไลน์ 100%** ไม่พึ่งอินเทอร์เน็ต —
  เหมาะกับงานราชการที่บางเครื่องไม่ต่อเน็ต
- **ฟอนต์:** TH Sarabun New ทุก TextRun ต้องระบุ `font: "TH Sarabun New"`
- **ขนาดตัวอักษรมาตรฐาน (หน่วย half-point ของ docx.js, size = pt×2):**
  - หัวเรื่อง "บันทึกข้อความ": 29pt ตัวหนา กึ่งกลาง → `size: 58`
  - ป้ายกำกับ (ส่วนราชการ/ที่/วันที่/เรื่อง): 20pt ตัวหนา → `size: 40`
  - เนื้อหาทั่วไป (เรียน/ภาคเหตุ/ภาคความประสงค์/ภาคสรุป/ผู้ลงชื่อ): 16pt ปกติ → `size: 32`
- **ระยะขอบกระดาษ:** บน/ล่าง 20mm, ซ้าย 30mm (เผื่อเจาะแฟ้ม), ขวา 20mm
- **ตราครุฑ:** ใส่เป็น floating ImageRun ขนาด ~1.5×1.5cm (57×57px ที่ 96dpi)
  วางห่างขอบบน 1.5cm และห่างขอบซ้าย 2.5cm โดยอิง `HorizontalPositionRelativeFrom.PAGE` /
  `VerticalPositionRelativeFrom.PAGE` และแปลงหน่วยด้วย `convertMillimetersToTwip(mm) * 635`
  (1 twip = 635 EMU) ผู้ใช้อัปโหลดรูปตราครุฑเองผ่าน `<input type="file">` — ถ้าไม่อัปโหลด
  จะเว้นกรอบไว้ในพรีวิว แต่ **จะไม่แทรกรูปใดๆ ลงไฟล์ .docx จริง** (ต้องเติมเองภายหลังถ้าจำเป็น)
- **การจัดบรรทัด ที่/วันที่:** ใช้ `tabStops` แบบ `TabStopType.LEFT` (ไม่ใช้ dot leader
  เพราะฟอร์มนี้กรอกค่าจริงจาก input ไม่ใช่เว้นบรรทัดว่างให้เขียนมือ) — ถ้าเอกสารประเภทอื่น
  ต้องการเว้นบรรทัดจุดไข่ปลาให้กรอกมือ ให้ใช้ `TabStopType.LEFT` + `leader: LeaderType.DOT`
  (เทคนิคนี้เคยใช้ได้ผลกับ LibreOffice PDF conversion มาก่อน ส่วน `PositionalTab` ใช้ไม่ได้)
- **วันที่:** แปลงจาก `<input type="date">` (ค.ศ.) เป็น พ.ศ. + ชื่อเดือนไทยเต็ม ด้วยฟังก์ชัน
  `toThaiDate()` ใน index.html แต่ละไฟล์ (ยังไม่ได้แยกเป็น shared helper — ถ้าเพิ่มโมดูลใหม่
  ควรย้าย logic นี้ไปไว้ที่ `shared/thai-utils.js` แทนการก็อปวาง)

## สถานะปัจจุบัน

- ✅ `บันทึกข้อความ/index.html` — บันทึกข้อความทั่วไป ใช้งานได้ ทดสอบสร้างไฟล์ .docx จริงแล้ว
  (ตรวจ zip/xml ถูกต้อง, มี `<w:drawing>`/`wp:anchor` เมื่อแนบรูปตราครุฑ) มีฟิลด์: ส่วนราชการ,
  ที่, วันที่, เรื่อง, เรียน, สิ่งที่ส่งมาด้วย (ไม่บังคับ), ภาคเหตุ, ภาคความประสงค์,
  ภาคสรุป (ไม่บังคับ), ผู้ลงชื่อ, ตำแหน่ง, อัปโหลดตราครุฑ + มีพรีวิวสดด้านขวา

- ✅ `ขออนุมัติโครงการก่อสร้าง/index.html` — บันทึกข้อความเสนอขออนุมัติจัดทำโครงการก่อสร้าง
  แปลงมาจากไฟล์ตัวอย่างจริงที่ผู้ใช้ส่งมา (`1. อนุมัติโครงการ.doc` — อบต.คลองด่าน)
  โครงสร้าง: หัวบันทึก (ส่วนราชการ/ที่/วันที่/เรื่อง/เรียน) → เรื่องเดิม → ข้อเท็จจริง →
  ข้อกฎหมายและระเบียบปฏิบัติ (พรีฟิลบทมาตรฐาน พ.ร.บ.จัดซื้อจัดจ้างฯ พ.ศ. ๒๕๖๐ / ระเบียบ
  กระทรวงการคลังฯ / หนังสือ มท ๐๘๐๘.๒) → ข้อพิจารณา (พรีฟิล) → สรุป/ข้อเสนอ (พรีฟิล) →
  ลงชื่อ 3 ระดับ: ผู้เสนอ (หน.กองช่าง) / ปลัด / นายก พร้อมช่องกาเครื่องหมาย
  "☐ อนุมัติ ☐ ไม่อนุมัติ" ท้ายเอกสาร ทดสอบสร้าง .docx ผ่าน Node sandbox แล้ว ใช้งานได้

- ✅ `คำสั่งแต่งตั้งคณะกรรมการ/index.html` — คำสั่งแต่งตั้งคณะกรรมการจัดทำแบบรูปรายการ
  งานก่อสร้าง แปลงมาจาก `2. คำสั่งจัดทำรูปแบบ.docx` (อบต.คลองด่าน) โครงสร้าง: หัวคำสั่ง
  (คำสั่ง{หน่วยงาน}/ที่/เรื่อง/ชื่อโครงการ กึ่งกลางตัวหนา + เส้นประคั่น) → ความนำ 2 ย่อหน้า
  (พรีฟิลเทมเพลต แก้ไขตามโครงการจริง) → **รายชื่อคณะกรรมการแบบไดนามิก** (เพิ่ม/ลบแถวได้
  ในฟอร์ม แต่ละแถว: ยศ/ชื่อ-สกุล, ตำแหน่ง, หน้าที่ [ประธานกรรมการ/กรรมการ/กรรมการและ
  เลขานุการ] — เรนเดอร์เป็นบรรทัดคั่นด้วย tabStops 3 จุด) → หน้าที่คณะกรรมการ (พรีฟิล) →
  วันที่สั่ง (date picker → แปลงเป็น "สั่ง ณ วันที่ ... เดือน ... พ.ศ. ...") → ลงชื่อนายก
  ตราครุฑในโมดูลนี้ใช้ตำแหน่ง**กึ่งกลางบนหัวกระดาษ** (`HorizontalPositionAlign.CENTER`)
  ต่างจากโมดูลบันทึกข้อความที่วางชิดซ้ายบน — เพราะแบบฟอร์มคำสั่งต้นฉบับวางครุฑกึ่งกลาง
  ทดสอบสร้าง .docx ผ่าน Node sandbox แล้ว (มี wp:anchor + ข้อความกรรมการถูกต้อง)

**ไฟล์ต้นฉบับที่ใช้อ้างอิง:** ผู้ใช้ส่งไฟล์ตัวอย่างจริงของ อบต.คลองด่าน (`1. อนุมัติโครงการ.doc`,
`2. คำสั่งจัดทำรูปแบบ.docx`) ไม่ได้เก็บสำเนาไว้ใน ConsDocx — ถ้าจะปรับ template เพิ่มเติม
ต้องขอผู้ใช้ส่งไฟล์เดิมมาอีกครั้ง หมายเหตุ:ต้นฉบับพิมพ์ผิด "บักทึกข้อความ" (แก้เป็น
"บันทึกข้อความ" ในโมดูลแล้ว) และใช้ฟอนต์ชื่อ "TH SarabunIT๙"/"TH SarabunPSK" ซึ่งเป็นคนละชื่อ
กับมาตรฐาน "TH Sarabun New" ที่โปรเจกต์นี้ยึดใช้ทุกโมดูล (ขนาดตัวอักษรที่วัดได้จาก XML จริง
ตรงกับที่ตั้งไว้แล้ว: หัวเรื่อง 29pt / ป้ายกำกับ 20pt / เนื้อหา 16pt)

## รอบแก้ไข: ระยะขอบ/ย่อหน้า + ฝังตราครุฑอัตโนมัติ (2026-07-17 รอบ 2)

ผู้ใช้แจ้งว่าระยะต่างๆ ยังไม่ตรงกับแบบฟอร์มต้นฉบับ และเซฟรูปตราครุฑ 2 ไฟล์ไว้ในโฟลเดอร์
(`krut-1.5-cm.png`, `krut-3-cm.png` — ตั้งชื่อตามความสูงจริงของครุฑที่ต้องการ ไม่ใช่ขนาดไฟล์)
ได้แก้ไขดังนี้:

- **วัดค่าจริงจากไฟล์ต้นฉบับด้วย python-docx** แทนการประมาณ: margin บน = **2.5 ซม.** (เดิมตั้งไว้
  2 ซม. — ผิด) margin ล่าง/ซ้าย/ขวา (2/3/2 ซม.) ถูกต้องอยู่แล้ว ย่อหน้าแรกของเนื้อหาในต้นฉบับ
  ใช้ 2 ตัว tab (default tab = 1.27 ซม. → รวม 2.54 ซม.) จึงปรับ `indent.firstLine` จาก 1.25 ซม.
  เป็น **2.5 ซม.** ในทั้ง 3 โมดูล (ตรงกับหลักเกณฑ์ย่อหน้าเอกสารราชการทั่วไปด้วย)

- **ฝังตราครุฑอัตโนมัติ**: แปลงรูปทั้งสองไฟล์เป็น base64 เก็บไว้ที่ `shared/crests.js`
  (`window.CRESTS.small` = krut-1.5-cm.png, `window.CRESTS.large` = krut-3-cm.png) โหลดผ่าน
  `<script src="../shared/crests.js">` เหมือน docx.lib.js — **ไม่ใช้ fetch()** เพราะเปิดไฟล์แบบ
  file:// ตรงๆ จะติด CORS แต่ script tag โหลดไฟล์ local ได้ปกติ ทุกโมดูลตอนนี้ใส่ตราครุฑให้
  อัตโนมัติตั้งแต่เปิดหน้า (ไม่ต้องอัปโหลดเอง) ช่องอัปโหลดยังอยู่เป็นตัวเลือกสำรอง/เผื่อเปลี่ยนภาพ
  ขนาดจริงคำนวณจาก naturalWidth/naturalHeight ของรูป (ผ่าน `Image.onload`) เทียบสัดส่วนกับ
  ความสูงเป้าหมาย (1.5 ซม. หรือ 3 ซม.) เพื่อไม่ให้ภาพบิดเบี้ยว:
  - `krut-1.5-cm.png` จริง 83×88px → ใช้ในโมดูล **บันทึกข้อความ** และ **ขออนุมัติโครงการก่อสร้าง**
    (ทั้งคู่เป็นบันทึกข้อความ) วางตำแหน่งชิดซ้ายบน ห่างขอบบน/ซ้าย 2.5 ซม. (ตรงกับที่ตราครุฑ
    ปรากฏข้างหัวเรื่อง "บันทึกข้อความ" บรรทัดเดียวกันในต้นฉบับจริง — ยืนยันด้วยการ render PDF เทียบ)
  - `krut-3-cm.png` จริง 659×738px → ใช้ในโมดูล **คำสั่งแต่งตั้งคณะกรรมการ** วางกึ่งกลางบนกระดาษ
    ห่างขอบบน 2.5 ซม. **แก้บั๊กจากรอบก่อน**: โค้ดเดิมใช้ key `alignment` ผิด (ต้องเป็น `align`)
    ทำให้ครุฑไม่ได้อยู่กึ่งกลางจริง ๆ — แก้แล้ว และเพิ่ม spacing ก่อนหัวเรื่อง "คำสั่ง..." เป็น
    3 ซม. (`mm(30)`) เพื่อให้ตัวหนังสือไม่ทับครุฑขนาดใหญ่ (สูตรคำนวณ: offset บน 2.5 ซม. + สูงครุฑ
    3 ซม. = ขอบล่างครุฑอยู่ที่ 5.5 ซม. จากขอบกระดาษ, margin บน 2.5 ซม. → ต้องเว้นเพิ่มอีก 3 ซม.
    ก่อนบรรทัดแรกของเนื้อหา)

- ตรวจสอบผลลัพธ์จริงด้วยการสร้าง .docx ทดสอบ → แปลง PDF ด้วย LibreOffice → render เป็น PNG ที่
  120dpi เทียบด้วยตา ยืนยันว่าครุฑอยู่ตำแหน่งถูกต้องและไม่ทับข้อความแล้วทั้งสองแบบ

- ไฟล์รูปต้นฉบับที่ผู้ใช้เซฟไว้ (`ConsDocx/krut-1.5-cm.png`, `ConsDocx/krut-3-cm.png`) ยังอยู่ที่
  root เหมือนเดิม ไม่ได้ลบ — คัดลอกสำเนาเข้า `shared/` เพิ่มเพื่อแปลงเป็น `crests.js` เท่านั้น

## เปลี่ยนสถาปัตยกรรม: จาก "สร้างเอกสารด้วยโค้ดล้วน" → "template + แทนที่คำ" (2026-07-17 รอบ 3)

ผู้ใช้เสนอแนวคิด (และผมเห็นด้วยว่าดีกว่า): แทนที่จะให้ docx.js สร้างเอกสารทั้งฉบับด้วยโค้ด
(ต้องเขียน margin/indent/tab เองทุกจุด แก้ยาก) ให้ใช้ไฟล์ **template.docx** ที่จัดฟอร์แมต
ถูกต้องแล้ว (แก้ไขได้ตรงๆ ด้วย Word) เป็นต้นแบบ แล้วโปรแกรมแค่ **แทนที่ placeholder** เช่น
`[ส่วนราชการ]` `[เรื่อง]` ด้วยข้อมูลที่กรอก แล้ว "บันทึกเป็น" ไฟล์ใหม่ ไม่แก้ template เดิม
(ตรงกับ workflow ที่ผู้ใช้อธิบาย: 1. มี template → 2. โปรแกรมแทนที่คำ → 3. save เป็นไฟล์ใหม่
โดย template จะเปลี่ยนก็ต่อเมื่อผู้ใช้แก้เองเท่านั้น)

**สถานะ: ทำ proof-of-concept เสร็จแล้วสำหรับโมดูล `บันทึกข้อความ/` เท่านั้น** (โมดูลอื่นยังเป็น
docx.js แบบเดิม รอดูผลตอบรับจากผู้ใช้ก่อนแปลงที่เหลือ)

### กลไกทำงาน
- `บันทึกข้อความ/template.docx` — ไฟล์ Word จริง เปิดแก้ไขได้ตามปกติ มีตราครุฑฝังอยู่ในไฟล์แล้ว
  (สร้างจาก docx.js ครั้งเดียวโดยใส่ placeholder แทนค่าจริง แล้ว "แช่แข็ง" เป็น .docx ต้นแบบ)
- `บันทึกข้อความ/template-data.js` — เนื้อไฟล์ template.docx แปลงเป็น base64 ฝังในตัวแปร
  `window.TEMPLATE_DOCX_BASE64` โหลดผ่าน `<script src="./template-data.js">` **ไม่ใช้ fetch()**
  เพราะเปิดผ่าน file:// จะติด CORS (เหตุผลเดียวกับที่ docx.lib.js/crests.js ใช้วิธีนี้)
- `shared/jszip.lib.js` — JSZip (unpkg dist ตัวเต็ม ~96KB) ใช้เปิด/แก้/บีบ .docx (เป็น zip) ใน
  เบราว์เซอร์ ใช้ร่วมกันได้ทุกโมดูลที่จะแปลงเป็น template-based ในอนาคต
- ตอนสร้างไฟล์: โหลด template (จาก base64 หรือจากไฟล์ที่ผู้ใช้อัปโหลดแทนถ้าแก้ template เอง) →
  `JSZip.loadAsync` → อ่าน `word/document.xml` เป็น string → แทนที่ทุก placeholder ด้วยฟังก์ชัน
  `replacePlaceholder()` → เขียนกลับ zip → `generateAsync({type:"blob"})` → ดาวน์โหลด

### จุดสำคัญที่ทำให้ใช้งานได้จริง (ไม่ใช่แค่ทฤษฎี)
- **Placeholder ต้องอยู่ครบในรันเดียว (run เดียว) ในไฟล์ template** — เกิดขึ้นเองเพราะ template
  สร้างจาก docx.js (เราคุมทุก run เอง) ถ้าผู้ใช้พิมพ์ placeholder ใหม่ใน Word เองต้อง **พิมพ์รวด
  เดียวไม่สลับฟอร์แมตกลางคำ** (ตัวหนา/ธรรมดา) ไม่งั้น Word จะตัดคำเป็นหลายรันและค้นหาไม่เจอ
- **`replacePlaceholder()` รองรับ placeholder ที่อยู่ปนกับข้อความอื่นในรันเดียวกัน** เช่น
  `"ที่  [ที่]"` หรือ `"([ผู้ลงชื่อ])"` — เวอร์ชันแรกที่เขียนตรวจแค่ `<w:t>[ที่]</w:t>`
  แบบ exact-match แล้วพลาด (หา `[ที่]` `[วันที่]` `[ผู้ลงชื่อ]` `[ตำแหน่ง]` ไม่เจอเพราะมันอยู่
  ปนกับข้อความอื่นในรันเดียวกัน) แก้เป็นจับ regex ที่ยอมให้มี prefix/suffix ในข้อความเดียวกัน
  แล้วตัด prefix/suffix ออกมาประกอบกลับหลังแทนค่า — ทดสอบผ่านแล้วด้วย Node+JSZip จริง
- **ค่าหลายบรรทัด** (เช่น ภาคเหตุที่มีข้อ 1. 2. 3.) แปลงเป็น `<w:br/>` ภายในรันเดียวกัน (ไม่ใช่
  ย่อหน้าใหม่) เพื่อให้ยังคง rPr/format เดิมของ placeholder นั้นได้ครบ
- ทดสอบ end-to-end จริง: สร้าง .docx ทดสอบ → ตรวจ zip ไม่เสีย → แปลง PDF ด้วย LibreOffice →
  render PNG เทียบด้วยตา → ครุฑ/ข้อความ/บรรทัดใหม่ถูกต้องหมด ไม่มี placeholder หลงเหลือ

### ข้อจำกัดที่ต้องบอกผู้ใช้ (ยังไม่ได้แก้)
- ฟิลด์ที่ไม่บังคับ (เช่น สิ่งที่ส่งมาด้วย, ภาคสรุป) ถ้าปล่อยว่างไว้ บรรทัดป้ายชื่อจะยังโชว์
  (เช่น "สิ่งที่ส่งมาด้วย" ลอยเปล่าๆ) เพราะการลบทั้งย่อหน้าออกเมื่อค่าว่างยังไม่ได้ทำ
  (ต้องลบทั้ง `<w:p>...</w:p>` ไม่ใช่แค่แทนที่ข้อความ — ซับซ้อนกว่านี้ ยังไม่ทำในรอบนี้)
- โมดูลที่มีรายการแบบไม่จำกัดจำนวน (เช่น รายชื่อคณะกรรมการใน "คำสั่งแต่งตั้งคณะกรรมการ")
  ยังไม่เหมาะกับวิธี template+แทนที่คำแบบง่ายนี้ เพราะจำนวนแถวไม่คงที่ ต้องมีกลไก "วนซ้ำ
  บล็อก" เพิ่มเติม (ยังไม่ได้ออกแบบ) — โมดูลนี้จึงยังคงใช้ docx.js แบบเดิม (ที่แก้ระยะ/ครุฑ
  ถูกต้องแล้วในรอบก่อนหน้า)
- ถ้าผู้ใช้แก้ไข template.docx เองแล้วอยากให้เป็นค่า default (ไม่ต้องอัปโหลดทุกครั้ง) ต้องขอให้
  ผมช่วย regenerate `template-data.js` ใหม่จากไฟล์ที่แก้แล้ว (เป็น manual step เพราะไฟล์ embed
  เป็น base64 ไว้ตอนนี้ — ระหว่างนี้ใช้ช่องอัปโหลด "ใช้ template.docx ฉบับที่แก้ไขเอง" แทนได้เลย
  โดยไม่ต้องรอผม)

## รอบที่ 4: เปลี่ยนไปใช้ "ไฟล์จริงของผู้ใช้" เป็น template โดยทำเครื่องหมายสีแดง (2026-07-17)

ผู้ใช้อยากให้ใช้ไฟล์ตัวเองเป็น template ตรงๆ (ไม่ใช่ไฟล์ที่ผมสร้างใหม่ด้วย docx.js) วิธีที่ตกลง
กัน: ผู้ใช้เปิด Word แล้วเปลี่ยนสีตัวอักษรเป็น**สีแดง**เฉพาะจุดที่ต้องการให้เป็นค่าตัวแปร ส่งไฟล์
กลับมา ผมสแกนหาข้อความสีแดงแล้วแปลงเป็น placeholder อัตโนมัติ

ผู้ใช้ส่งไฟล์ทดสอบมา 2 ไฟล์ (`1. อนุมัติโครงการ temp.doc`, `2. คำสั่งจัดทำรูปแบบ temp.docx`)
โดยทำสีแดงไว้**บางจุดเท่านั้น** (บอกไว้ว่า "ลองแค่นี้ก่อน" จะเพิ่มทีหลัง):
- เอกสาร 1 (บันทึกขออนุมัติ): แดงแค่ 2 จุด — ชื่อโครงการ (ในบรรทัดเรื่อง), ข้อความอ้างอิงแผนพัฒนาท้องถิ่น
- เอกสาร 2 (คำสั่ง): แดง 4 กลุ่ม — ชื่อโครงการ (ซ้ำ 3 จุดในเอกสาร), ชื่อ+ตำแหน่งกรรมการทั้ง 3 คน (6 placeholder)

**สคริปต์แปลง** (`/tmp/build_templates.py` — รันครั้งเดียว ไม่ได้เก็บไว้ถาวรใน ConsDocx เพราะเป็น
one-off, ถ้าต้องแปลงไฟล์ red-mark ใหม่ในอนาคตต้องเขียนใหม่หรือขอให้ผมทำอีกครั้ง):
ใช้ python-docx หาราน (run) ที่สีออกแดง (`R>=180, G<=90, B<=90` ครอบคลุมทั้ง `FF0000` และ `EE0000`
ที่เจอในไฟล์จริง) แล้วรวมกลุ่มรานสีแดงที่ติดกัน (บาง cell มีมากถึง ~24 รานย่อยเพราะ Word ตัดคำ
อัตโนมัติ) เป็นรานเดียว ใส่ placeholder `[ชื่อ...]` แทน แล้ว**เซ็ตสีกลับเป็นดำ `000000` อย่างชัดเจน**
(ตอนแรกแค่ลบ `<w:color>` ออกทำให้ LibreOffice render เป็นสีน้ำเงิน-ขีดเส้นใต้แบบ hyperlink โดยไม่ทราบ
สาเหตุแน่ชัด — แก้โดยตั้งค่า color เป็น 000000 ตรงๆ แทนการลบ)

**บั๊กสำคัญที่เจอและแก้ในฟังก์ชัน `replacePlaceholder()` (JS ฝั่งเบราว์เซอร์):**
1. Placeholder ที่ปนอยู่กับข้อความอื่นในรานเดียวกัน (เช่น `"ที่  [ที่]"`) — regex เดิมเช็ก exact-match
   `<w:t>[ที่]</w:t>` เท่านั้น หาไม่เจอ ต้องแก้เป็นจับ prefix/suffix รอบ placeholder ในข้อความเดียวกัน
2. **ไฟล์ Word จริงมี attribute ใน `<w:r>`** เช่น `<w:r w:rsidRPr="0034739A">` (ต่างจากไฟล์ที่ผมสร้างเอง
   ด้วย docx.js ซึ่งได้ `<w:r>` เปล่าๆ เสมอ) — regex เดิม hardcode `<w:r>` พอดีเป๊ะ เจอไฟล์จริงแล้วหาไม่เจอ
   เลยสักตัวเดียว ต้องแก้เป็น `<w:r[^>]*>` ในทั้ง 3 โมดูล (นี่คือเหตุผลที่ทดสอบกับไฟล์จริงสำคัญมาก —
   ไฟล์ที่ผมสร้างเองไม่เจอบั๊กนี้)
3. Placeholder ที่ซ้ำหลายจุดในเอกสารเดียว (เช่น `[ชื่อโครงการ]` ในคำสั่ง ซ้ำ 3 จุด) ต้องเปลี่ยนจาก
   `match()` ครั้งเดียวเป็น regex แบบ `g` flag วนลูปแทนที่ทุกจุดที่เจอ (สร้าง `result` string สะสม
   ทีละช่วงแทนการใช้ `.replace()` ตรงๆ เพราะต้องคง index ในการวนต่อ)

ทดสอบยืนยันด้วย Node+JSZip จริงทุกจุด: เอกสาร 2 มี `[ชื่อโครงการ]` 3 จุดถูกแทนที่ครบ, กรรมการ
ทั้ง 3 คนถูกแทนที่ถูกต้อง, แปลง PDF เทียบด้วยตาแล้วครุฑ/ระยะ/ฟอนต์ตรงกับไฟล์จริงเป๊ะ (เพราะใช้
ไฟล์จริงเป็นฐานตรงๆ ไม่ใช่ไฟล์จำลอง)

**โครงสร้างโมดูลตอนนี้ (หลังรอบนี้):**
- `บันทึกข้อความ/` — template สร้างเองด้วย docx.js (11 placeholder ครบทุกฟิลด์หลัก)
- `ขออนุมัติโครงการก่อสร้าง/` — **เปลี่ยนเป็น template จากไฟล์จริงแล้ว** มีแค่ 2 placeholder
  ([ชื่อโครงการ], [อ้างอิงแผนพัฒนาท้องถิ่น]) ฟิลด์อื่นคงที่ตามไฟล์ต้นฉบับ อบต.คลองด่าน
- `คำสั่งแต่งตั้งคณะกรรมการ/` — **เปลี่ยนเป็น template จากไฟล์จริงแล้ว** มี 7 placeholder
  ([ชื่อโครงการ] + ชื่อ/ตำแหน่งกรรมการ 3 คน) จำนวนกรรมการตายตัวที่ 3 คน (ไม่ใช่ dynamic add/remove
  แบบเดิมอีกต่อไป เพราะเปลี่ยนมาใช้ template-replace ที่ไม่รองรับจำนวนแถวไม่คงที่)
  เลขที่คำสั่ง/วันที่สั่ง/ชื่อนายกผู้ลงนาม ยังคงที่ตามไฟล์ต้นฉบับ (ยังไม่ได้ทำสีแดง)

**ขั้นต่อไปที่ผู้ใช้บอกไว้:** จะกลับไปทำสีแดงเพิ่มจุดที่เหลือ (ส่วนราชการ/ที่/วันที่/เรียน/งบประมาณ/
ชื่อผู้ลงนามต่างๆ ที่ยังไม่ได้ทำในเอกสาร 1, และเลขที่คำสั่ง/วันที่/ชื่อนายกในเอกสาร 2) แล้วส่งมาให้
แปลงเพิ่ม — เมื่อได้ไฟล์ใหม่ให้รันขั้นตอนเดิม (หา red run groups → merge → ใส่ placeholder → เซ็ต
สีดำ → แปลง base64 ใหม่ทับ template-data.js เดิม → เพิ่มฟิลด์ในฟอร์ม `index.html` ให้ตรง)

## รอบที่ 5: แก้ปัญหา "เว้นบรรทัดเพี้ยน" + "Thai Distributed" หายไป (2026-07-17)

ผู้ใช้แจ้งว่า "คำสั่ง" โอเคแล้ว แต่ "ขออนุมัติโครงการก่อสร้าง" มีปัญหาการเว้นบรรทัด/ระยะบรรทัด
และเรื่อง **Thai Distributed** (การจัดบรรทัดแบบกระจายอักษรไทย — ตัวเลือกจัดหน้าใน Word ที่ต่างจาก
"ชิดขอบ/กึ่งกลาง/ชิดซ้าย/ชิดขวา" ปกติ คือกระจายระยะห่างระหว่าง**ตัวอักษร**แทนที่จะกระจายแค่ที่
ช่องว่าง — จำเป็นสำหรับข้อความไทยเพราะปกติไม่มีช่องว่างระหว่างคำ ถ้าไม่ใช้โหมดนี้ข้อความยาวๆ ที่
ตัดขึ้นบรรทัดใหม่จะดูขอบขวาไม่เรียบ/เว้นบรรทัดแปลกๆ)

**สาเหตุที่หาเจอ:** ไฟล์ต้นฉบับ "1. อนุมัติโครงการ.doc" เป็น**ไฟล์ .doc รูปแบบเก่า** ต้องแปลงผ่าน
LibreOffice ก่อนถึงจะเปิดเป็น .docx ให้โปรแกรมแก้ไขได้ — ขั้นตอนแปลงนี้เองที่ทำให้ค่า
`w:jc="thaiDistribute"` (Thai Distributed) ของทุกย่อหน้า**ถูกเขียนทับกลายเป็น `w:jc="start"`
(ชิดซ้ายธรรมดา)** โดย LibreOffice เอง — ตรวจสอบยืนยันแล้วโดยเทียบกับไฟล์ "2. คำสั่งจัดทำรูปแบบ"
ซึ่งเป็น **.docx อยู่แล้วตั้งแต่ต้น ไม่ต้องผ่าน LibreOffice แปลง** จึงมี `thaiDistribute` อยู่ครบ
9 ย่อหน้าไม่มีปัญหา (ตรงกับที่ผู้ใช้บอกว่าไฟล์นี้โอเคอยู่ — ยืนยัน root cause ตรงเป๊ะ)

**วิธีแก้ที่ทำไปแล้ว:** เปิด `template.docx` ของโมดูล "ขออนุมัติโครงการก่อสร้าง" ด้วย python
แก้ `<w:jc w:val="start"/>` เป็น `<w:jc w:val="thaiDistribute"/>` ทั้งหมด (17 จุด) โดยตรงในระดับ
XML (ไม่ผ่าน LibreOffice ซ้ำ เพื่อไม่ให้เจอปัญหาเดิมอีก) แล้วสร้าง `template-data.js` (base64) ใหม่
ทดสอบแทนที่ placeholder ด้วย JSZip อีกครั้งว่ายังทำงานถูกต้อง (ไม่มี placeholder ตกค้าง) — แต่
**ยืนยันผลด้วยตาไม่ได้ในแซนด์บ็อกซ์นี้** เพราะไม่มีฟอนต์ TH Sarabun New ติดตั้งอยู่ (LibreOffice
render เป็นกล่องสี่เหลี่ยมแทนตัวอักษรไทย มองไม่เห็นผลต่างระหว่าง start กับ thaiDistribute) —
**ต้องให้ผู้ใช้เปิดไฟล์บนเครื่องตัวเองเพื่อยืนยันว่าเรียบร้อยแล้วจริง**

**ทางเลือกสำหรับอนาคตถ้ายังไม่เรียบร้อย หรือจะแปลง .doc ไฟล์อื่นอีก:** ถ้าผู้ใช้มี Microsoft Word
ในเครื่อง แนะนำให้เปิดไฟล์ .doc ต้นฉบับด้วย Word จริง (ไม่ใช่ LibreOffice) แล้ว "Save As" เป็น .docx
เอง ก่อนทำเครื่องหมายสีแดง — Word จะแปลงฟอร์แมตของตัวเองได้แม่นยำกว่า LibreOffice ไม่เสี่ยงเจอ
ปัญหา thaiDistribute หายแบบนี้อีก (ไฟล์ .docx ที่เป็น native อยู่แล้วอย่าง "คำสั่ง" ไม่มีปัญหานี้เลย)

## รอบที่ 6: rebuild template จากไฟล์ .docx จริง (native) แทนไฟล์ที่ผ่าน LibreOffice แปลง (2026-07-17)

ผู้ใช้ทำตามคำแนะนำในรอบที่ 5 — เปิด "1. อนุมัติโครงการ.doc" ด้วย Microsoft Word จริง (ไม่ใช่
LibreOffice) แล้ว Save As เป็น .docx เอง พร้อมทำเครื่องหมายสีแดงใหม่ ส่งไฟล์ "1. อนุมัติโครงการ

**ตรวจสอบไฟล์ใหม่ก่อน rebuild:** เปิดด้วย python-docx/zipfile ยืนยันว่า `<w:jc>` มีค่าถูกต้องครบ

**สิ่งที่พบระหว่างตรวจสอบ (สำคัญ ต้องแจ้งผู้ใช้):** จุดสีแดงในบรรทัด "เรื่อง" ของไฟล์ใหม่นี้
**สั้นกว่าไฟล์เดิม** — ไฟล์เดิม (`1. อนุมัติโครงการ temp.doc`) ทำสีแดงครอบคลุม
`"ก่อสร้างถนน คสล.ซอยบ้านเจ๊จิตร หมู่ที่ 8"` (รวม "หมู่ที่ 8" ด้วย) แต่ไฟล์ใหม่นี้ทำสีแดงแค่
`"ก่อสร้างถนน คสล.ซอยบ้านเจ๊จิตร"` เท่านั้น — ส่วน `" หมู่ที่ 8"` เหลือเป็นข้อความคงที่ (ไม่แดง)
ต่อท้ายอัตโนมัติเสมอ **ผลคือ:** ถ้าผู้ใช้กรอกชื่อโครงการใหม่ที่มีหมู่บ้านอื่น (เช่น "หมู่ที่ 9")

จริงที่เจอในไฟล์นี้ (ไม่รวม "หมู่ที่ 8")** เพราะยึดหลักไม่เดาเกินจุดที่ทำเครื่องหมายไว้จริง —
ถ้าผู้ใช้ต้องการให้ "หมู่ที่ ..." เป็นตัวแปรด้วย ต้องทำสีแดงเพิ่มค่อยส่งไฟล์มาแปลงใหม่

จุดแดงที่ 2 (ย่อหน้าอ้างอิงแผนพัฒนาท้องถิ่น) เหมือนไฟล์เดิมทุกประการ ไม่มีการเปลี่ยนแปลง

**ขั้นตอนที่ทำ:** รัน python-docx script เดิม (หา red run groups → merge เป็น placeholder →
เซ็ตสีดำ 000000) กับไฟล์ native .docx ใหม่นี้โดยตรง → ได้ `template.docx` ใหม่ (38,484 bytes)

ไม่มี placeholder ตกค้าง และ jc count หลังแทนที่ยังครบ 17+3 เหมือนเดิม → คัดลอกทับ
`template.docx`/`template-data.js` เดิมทั้งใน ConsDocx และ outputs mirror

**สถานะล่าสุด:** โมดูล "ขออนุมัติโครงการก่อสร้าง" ตอนนี้ใช้ template จากไฟล์ native .docx ที่ถูกต้อง

Sarabun New) ต้องรอผู้ใช้เปิดไฟล์ที่สร้างจากฟอร์มนี้เพื่อยืนยัน

## รอบที่ 7: แยกโฟลเดอร์ `templates/` เก็บไฟล์ template.docx ต้นฉบับกลาง (2026-07-17)

ผู้ใช้ขอให้แยกโฟลเดอร์เก็บ template.docx ต่างหาก (แทนที่จะฝังอยู่ในโฟลเดอร์โมดูลแต่ละอัน)
ย้ายไฟล์ `template.docx` ของทั้ง 3 โมดูลออกมาไว้ที่ `ConsDocx/templates/<ชื่อโมดูล>/template.docx`
โครงสร้างใหม่:

```
ConsDocx/
├── memory.md
├── shared/                          ← lib กลาง (docx.lib.js, jszip.lib.js, crests.js)
├── templates/                       ← ⭐ ใหม่: เก็บไฟล์ template.docx ต้นฉบับของทุกโมดูล
│   ├── บันทึกข้อความ/template.docx
│   ├── ขออนุมัติโครงการก่อสร้าง/template.docx
│   └── คำสั่งแต่งตั้งคณะกรรมการ/template.docx
├── บันทึกข้อความ/                    ← index.html + template-data.js (ไม่มี template.docx แล้ว)
├── ขออนุมัติโครงการก่อสร้าง/
└── คำสั่งแต่งตั้งคณะกรรมการ/
```

**หมายเหตุสำคัญ:** `template-data.js` (base64 ที่โปรแกรมโหลดใช้งานจริงตอนรัน) **ยังอยู่ที่เดิม**
ในโฟลเดอร์โมดูลแต่ละอัน — ไม่กระทบการทำงานของฟอร์มเลย เพราะ index.html ไม่เคยอ่าน template.docx
ตรงๆ (อ่านแต่ `window.TEMPLATE_DOCX_BASE64` จาก template-data.js) ไฟล์ `template.docx` จริงมีไว้
**สำหรับผู้ใช้เปิดดู/แก้ไขเองด้วย Word เท่านั้น** (เช่น จะทำสีแดงเพิ่ม) — ย้ายที่เก็บได้อย่างปลอดภัย
ไม่ต้องแก้โค้ดใดๆ ในการโหลดไฟล์

แก้ข้อความอ้างอิงตำแหน่งไฟล์ในกล่อง "template-box" ของทั้ง 3 `index.html` ให้ชี้ไปที่
`../templates/<ชื่อโมดูล>/` แทนคำว่า "ในโฟลเดอร์นี้" (เดิมอ้างผิดที่แล้วหลังย้ายไฟล์)

**กติกาเพิ่มเติมสำหรับโมดูลใหม่ในอนาคต:** เวลาสร้างโมดูลใหม่ ให้เก็บ `template.docx` ต้นฉบับไว้ที่
`templates/<ชื่อโมดูลใหม่>/template.docx` ตั้งแต่แรกเลย (ไม่ต้องเก็บซ้ำในโฟลเดอร์โมดูล)

## รอบที่ 8: แยก html ออกเป็นโฟลเดอร์ `forms/` + ตั้งชื่อ template ให้ตรงหัวข้อ (2026-07-17)

ผู้ใช้ขอให้แยก `index.html`/`template-data.js` ออกมาเก็บอีกโฟลเดอร์ต่างหาก (ไม่ปนกับโฟลเดอร์เดิม)
และเปลี่ยนชื่อไฟล์ `template.docx` ให้ตรงกับหัวข้อแต่ละโมดูล (เดิมทุกโมดูลใช้ชื่อ "template.docx"
เหมือนกันหมด เปิดพร้อมกันหลายไฟล์ใน Word แล้วสับสนว่าไฟล์ไหนของโมดูลไหน)

**โครงสร้างใหม่ล่าสุด:**

```
ConsDocx/
├── memory.md
├── shared/                                        ← lib กลาง (docx.lib.js, jszip.lib.js, crests.js)
├── templates/                                     ← ไฟล์ .docx ต้นแบบ (ชื่อไฟล์ตรงหัวข้อ ไม่ซ้ำกัน)
│   ├── บันทึกข้อความ_template.docx
│   ├── ขออนุมัติโครงการก่อสร้าง_template.docx
│   └── คำสั่งแต่งตั้งคณะกรรมการ_template.docx
└── forms/                                         ← ⭐ ใหม่: index.html + template-data.js ของทุกโมดูล
    ├── บันทึกข้อความ/
    │   ├── index.html
    │   └── template-data.js
    ├── ขออนุมัติโครงการก่อสร้าง/
    │   ├── index.html
    │   └── template-data.js
    └── คำสั่งแต่งตั้งคณะกรรมการ/
        ├── index.html
        └── template-data.js
```

**สิ่งที่แก้ไปด้วย (เพราะย้ายไฟล์ลึกขึ้นอีกชั้น):**
- path ของ `<script src="../shared/jszip.lib.js">` ในทั้ง 3 `index.html` แก้เป็น
  `../../shared/jszip.lib.js` (เพราะตอนนี้ index.html อยู่ลึกกว่าเดิม 1 ชั้น คือ
  `ConsDocx/forms/<โมดูล>/index.html` แทนที่จะเป็น `ConsDocx/<โมดูล>/index.html`)
- ข้อความในกล่อง "template-box" ของแต่ละ index.html แก้ให้ตรงชื่อไฟล์ใหม่และ path ใหม่
  (`../../templates/<ชื่อไฟล์>_template.docx`)
- `template-data.js` **ไม่ต้อง regenerate ใหม่** เพราะเนื้อหา base64 เดิมไม่เปลี่ยน (แค่ย้ายที่เก็บไฟล์
  .js เอง ไม่ได้แก้เนื้อหาข้างใน)

**โฟลเดอร์เดิม 3 อัน** (`ConsDocx/บันทึกข้อความ/`, `ConsDocx/ขออนุมัติโครงการก่อสร้าง/`,
`ConsDocx/คำสั่งแต่งตั้งคณะกรรมการ/`) **ลบไม่ได้** (ระบบไฟล์ไม่ยอมให้ลบโฟลเดอร์ที่เคยเขียนแล้ว)
เลยเหลือว่างอยู่ พร้อมไฟล์ข้อความสั้นๆ อธิบายว่าย้ายไปไหนแล้ว ("ย้ายไปที่ forms แล้ว.txt") — ผู้ใช้
ลบโฟลเดอร์เหล่านี้เองได้จากเครื่องถ้าต้องการให้สะอาดขึ้น (ไม่กระทบการทำงานของโปรแกรมเลย)

**กติกาเพิ่มเติมสำหรับโมดูลใหม่ในอนาคต:** โมดูลใหม่ให้วาง `index.html`+`template-data.js` ที่
`forms/<ชื่อโมดูลใหม่>/` และ `template.docx` ต้นฉบับที่ `templates/<ชื่อโมดูลใหม่>_template.docx`
ตั้งแต่แรกเลย (script src ต้องใช้ `../../shared/...` เพราะลึก 2 ชั้นจาก root)

## รอบที่ 9: ใส่เลขลำดับให้ template/form ตามลำดับงานจริง (2026-07-17)

ผู้ใช้แจ้งว่าจะทยอยส่งเอกสารเพิ่มอีกหลายลำดับ ขอให้ใส่เลขกำกับไว้ในชื่อไฟล์ template และ
ชื่อโมดูล html ตั้งแต่ตอนนี้ ตามลำดับงานจริงในโครงการก่อสร้าง (บอกไว้คร่าวๆ ว่า
"1. อนุมัติโครงการ 2. คำสั่งแบบรูปรายการ") — ใส่เลขให้ 2 โมดูลนี้แล้ว:

- `ขออนุมัติโครงการก่อสร้าง` → เปลี่ยนชื่อเป็น **`1. ขออนุมัติโครงการก่อสร้าง`**
- `คำสั่งแต่งตั้งคณะกรรมการ` → เปลี่ยนชื่อเป็น **`2. คำสั่งแต่งตั้งคณะกรรมการ`**

**`บันทึกข้อความ` (โมดูลทั่วไป) ยังไม่ใส่เลข** เพราะเป็นแบบฟอร์มเอนกประสงค์ ไม่ใช่ขั้นตอนเฉพาะ
ในลำดับงานก่อสร้าง (ผู้ใช้ไม่ได้พูดถึงตอนสั่งใส่เลข) — ถ้าอยากให้มีเลขด้วยหรือจัดลำดับใหม่
บอกได้ตลอด

**โครงสร้างหลังเปลี่ยน:**
```
ConsDocx/
├── templates/
│   ├── 1. ขออนุมัติโครงการก่อสร้าง_template.docx
│   ├── 2. คำสั่งแต่งตั้งคณะกรรมการ_template.docx
│   └── บันทึกข้อความ_template.docx          (ไม่มีเลข — เอนกประสงค์)
└── forms/
    ├── 1. ขออนุมัติโครงการก่อสร้าง/
    ├── 2. คำสั่งแต่งตั้งคณะกรรมการ/
    └── บันทึกข้อความ/
```

แก้ `<title>`, `<h1>`, และข้อความอ้างอิงชื่อไฟล์ template ในกล่อง "template-box" ของทั้ง 2
`index.html` ให้มีเลขนำหน้าตรงกับชื่อโฟลเดอร์/ไฟล์ใหม่ด้วย — เนื้อหา `template-data.js` (base64)
ไม่ต้องแก้ เพราะแค่เปลี่ยนชื่อโฟลเดอร์/ไฟล์ ไม่ได้แก้เนื้อหาไฟล์เอง (การ rename ไม่กระทบ
`<script src="./template-data.js">` เพราะเป็น path สัมพัทธ์ในโฟลเดอร์เดียวกัน)

**กติกาสำหรับโมดูลใหม่ที่จะเพิ่มต่อจากนี้:** ตั้งชื่อโฟลเดอร์ `forms/<เลข>. <ชื่อเรื่อง>/` และไฟล์
`templates/<เลข>. <ชื่อเรื่อง>_template.docx` ตามลำดับขั้นตอนจริงของงานก่อสร้าง (ผู้ใช้จะบอกเลข/ลำดับ
มาเรื่อยๆ เมื่อส่งไฟล์ตัวอย่างใหม่)

## รอบที่ 10: เปลี่ยนชื่อไฟล์ index.html ให้ตรงหัวข้อแต่ละโมดูล (2026-07-17)

ผู้ใช้ถามว่าไฟล์ `index.html` ชื่อซ้ำกันหมดทุกโฟลเดอร์ (แค่คนละโฟลเดอร์) จะสับสนไหม — ถามกลับ
ให้เลือกระหว่าง "เปลี่ยนชื่อให้ตรงหัวข้อ" กับ "คงชื่อ index.html ไว้" ผู้ใช้เลือก **เปลี่ยนชื่อให้ตรงหัวข้อ**

เปลี่ยนชื่อไฟล์ html ในทั้ง 3 โฟลเดอร์ (ชื่อไฟล์ตรงกับชื่อโฟลเดอร์เป๊ะ รวมเลขลำดับด้วย):
- `forms/1. ขออนุมัติโครงการก่อสร้าง/index.html` → **`1. ขออนุมัติโครงการก่อสร้าง.html`**
- `forms/2. คำสั่งแต่งตั้งคณะกรรมการ/index.html` → **`2. คำสั่งแต่งตั้งคณะกรรมการ.html`**
- `forms/บันทึกข้อความ/index.html` → **`บันทึกข้อความ.html`**

ไม่กระทบการทำงานเลย เพราะ `<script src="./template-data.js">` เป็น path สัมพัทธ์กับโฟลเดอร์
ไม่ใช่ชื่อไฟล์ html — เปลี่ยนแค่ชื่อไฟล์ ไม่ต้อง regenerate อะไรเพิ่ม

**โครงสร้างล่าสุด:**
```
ConsDocx/
├── memory.md
├── shared/
├── templates/
│   ├── 1. ขออนุมัติโครงการก่อสร้าง_template.docx
│   ├── 2. คำสั่งแต่งตั้งคณะกรรมการ_template.docx
│   └── บันทึกข้อความ_template.docx
└── forms/
    ├── 1. ขออนุมัติโครงการก่อสร้าง/
    │   ├── 1. ขออนุมัติโครงการก่อสร้าง.html   ← เปิดไฟล์นี้เพื่อใช้งาน
    │   └── template-data.js
    ├── 2. คำสั่งแต่งตั้งคณะกรรมการ/
    │   ├── 2. คำสั่งแต่งตั้งคณะกรรมการ.html
    │   └── template-data.js
    └── บันทึกข้อความ/
        ├── บันทึกข้อความ.html
        └── template-data.js
```

**กติกาเพิ่มเติมสำหรับโมดูลใหม่:** ตั้งชื่อไฟล์ html ให้ตรงกับชื่อโฟลเดอร์เป๊ะทุกครั้ง (รวมเลขลำดับ
ถ้ามี) แทนการใช้ "index.html" แบบเดิม

## รอบที่ 11: สร้างหน้า Main เชื่อมทุกเอกสาร + ส่งต่อฟิลด์ร่วม (2026-07-17)

ผู้ใช้เสนอแนวคิด: อยากมีหน้า HTML หลัก ("main") ที่ลิงก์ไปเอกสารแต่ละลำดับ โดยฟิลด์ที่ใช้ร่วมกัน
หลายเอกสาร (เช่น ชื่อโครงการ) กรอกที่หน้า main ครั้งเดียวพอ ส่วนฟิลด์เฉพาะของเอกสารนั้นๆ ยังเข้าไป
แก้ในหน้าเอกสารนั้นโดยตรงตามปกติ — ถามกลับว่าฟิลด์ไหนควรเป็น "ฟิลด์ร่วม" ผู้ใช้เลือก **ชื่อโครงการ**
และ **ผู้ลงนาม (นายก/ปลัด/หัวหน้ากอง)** เป็นหลัก (ยังไม่ระบุ "อื่นๆ" ชัดเจน) แล้วบอกให้ลองทำ main
ให้ดูก่อน

**สิ่งที่ทำในรอบนี้ (แค่ ชื่อโครงการ เท่านั้น เพราะพร้อมใช้งานจริง):**
- `ผู้ลงนาม (นายก/ปลัด/หัวหน้ากอง)` **ยังทำไม่ได้ตอนนี้** เพราะยังไม่เป็น placeholder ในทั้ง 2 template
  เลย (เป็นข้อความคงที่ ไม่เคยทำสีแดงไว้) — ต้องรอผู้ใช้ทำสีแดงที่ชื่อผู้ลงนามในไฟล์ template ก่อน
  ค่อยแปลงเป็น placeholder แล้วค่อยเพิ่มเข้าไปในฟิลด์ร่วมของหน้า main ทีหลัง
- `ชื่อโครงการ` เป็น placeholder อยู่แล้วทั้งในโมดูล 1 และ 2 (id="projectName" เหมือนกันทั้งคู่) —
  พร้อมเชื่อมได้ทันที

**กลไกที่ใช้ส่งต่อข้อมูล:** เลือกใช้ **URL query string** (เช่น
`?projectName=ก่อสร้างถนน...`) แทน `localStorage` เพราะไฟล์เปิดผ่าน `file://` โดยตรง (ไม่มีเซิร์ฟเวอร์)
และ browser บางตัว (โดยเฉพาะ Chrome) อาจแยก origin ของแต่ละไฟล์ file:// ออกจากกัน ทำให้
`localStorage` ไม่รับประกันว่าจะแชร์ข้ามไฟล์ .html กันได้เสมอ — ใช้ query string ผ่าน `<a href>` ที่สร้าง
แบบไดนามิกด้วย JS แทน ซึ่งเชื่อถือได้ 100% ไม่ขึ้นกับพฤติกรรม storage partitioning ของ browser

**ไฟล์ใหม่:** `ConsDocx/forms/main.html` — หน้าเมนูรวม มี:
1. กล่อง "ข้อมูลร่วม" ให้กรอก ชื่อโครงการ ครั้งเดียว
2. รายการลิงก์ไปเอกสารตามลำดับ (1. ขออนุมัติโครงการก่อสร้าง, 2. คำสั่งแต่งตั้งคณะกรรมการ,
   บันทึกข้อความทั่วไป) — ลิงก์ของเอกสาร 1 และ 2 จะแนบ `?projectName=...` ต่อท้ายอัตโนมัติ
   (อัปเดตสดด้วย JS ทุกครั้งที่พิมพ์) ส่วนบันทึกข้อความทั่วไปไม่มี query param เพราะไม่มีฟิลด์
   ชื่อโครงการที่ตรงกัน

**สิ่งที่แก้ในหน้าเอกสารทั้ง 2 (`1. ขออนุมัติโครงการก่อสร้าง.html`, `2. คำสั่งแต่งตั้งคณะกรรมการ.html`):**
เพิ่มฟังก์ชัน `prefillFromMain()` ในสคริปต์ (รันก่อน `updatePreview()` ตัวแรก) — อ่านค่า
`projectName` จาก `URLSearchParams(location.search)` แล้วใส่ลงในช่อง `#projectName` ถ้ามีค่าส่งมา
ถ้าเปิดตรงๆ โดยไม่ผ่าน main (ไม่มี query string) ก็ทำงานปกติเหมือนเดิมทุกประการ — ยังแก้ไขค่าใน
ฟอร์มนั้นได้อิสระตามปกติหลังกรอกมาแล้วด้วย

**ข้อจำกัด/สิ่งที่ยังไม่ได้ทำ:**
- ฟิลด์ร่วมอื่นๆ ที่ผู้ใช้บอกว่าจะเพิ่ม (ผู้ลงนาม, "อื่นๆ" ที่ยังไม่ระบุ) รอผู้ใช้: (1) ทำสีแดงเพิ่มใน
  template ที่เกี่ยวข้องเพื่อสร้าง placeholder ก่อน (2) แจ้งชื่อฟิลด์ "อื่นๆ" ที่ต้องการเพิ่มให้ชัดเจน
- `main.html` ตอนนี้ยังไม่มีลิงก์ตัวเองใน `ConsDocx/` root (ต้องเข้าไปที่ `forms/main.html` โดยตรง)
  ถ้าต้องการให้เป็นทางเข้าโปรแกรมจากหน้าแรกสุด อาจสร้างไฟล์ `ConsDocx/index.html` ที่ redirect หรือ
  ลิงก์ไป `forms/main.html` อีกที (ยังไม่ได้ทำในรอบนี้)

## รอบที่ 12: เอาบันทึกข้อความออกจาก main + ปุ่มกลับหน้าแรกทุกหน้า + ปุ่มสร้างไฟล์ทั้งหมดเป็น zip (2026-07-17)

ผู้ใช้ขอ 3 อย่างพร้อมกัน:
1. เอาลิงก์ "บันทึกข้อความทั่วไป" ออกจากรายการเอกสารใน `main.html` (เหลือแค่เอกสาร 1 และ 2)
2. เพิ่มปุ่ม/ลิงก์ "กลับไปหน้าแรก" ใน **ทุก** html เอกสาร (รวมบันทึกข้อความทั่วไปด้วย แม้จะเอาออกจาก
   รายการใน main แล้วก็ตาม เพราะผู้ใช้ระบุ "ทุก html")
3. ที่หน้า main ให้มีปุ่มสั่งสร้างไฟล์ **ทั้งหมด** พร้อมกัน แล้วส่งออกรวมเป็นไฟล์ .zip เดียว

**สิ่งที่ทำ:**
- แก้ `forms/main.html`: ลบ `<li>` ของ "บันทึกข้อความทั่วไป" ออกจากรายการเอกสาร (เหลือ 2 รายการ)
- เพิ่มลิงก์ `&larr; กลับไปหน้าแรก` (ชี้ไป `../main.html`) ไว้บนสุดของทั้ง 3 หน้าเอกสาร
  (`1. ขออนุมัติโครงการก่อสร้าง.html`, `2. คำสั่งแต่งตั้งคณะกรรมการ.html`, `บันทึกข้อความ.html`)
- เพิ่มปุ่ม **"สร้างไฟล์ทั้งหมด (.zip)"** ในการ์ดใหม่ท้ายหน้า `main.html`

**ปัญหาทางเทคนิคที่ต้องแก้ก่อนทำปุ่มนี้ได้:** `template-data.js` ของทั้ง 2 โมดูลใช้ชื่อตัวแปร global
เดียวกันคือ `window.TEMPLATE_DOCX_BASE64` — ถ้าโหลดทั้งคู่พร้อมกันในหน้าเดียว (main.html) ตัวที่โหลด
ทีหลังจะทับตัวแปรของตัวแรก ใช้ไม่ได้ทั้งคู่พร้อมกัน **วิธีแก้:** สร้างไฟล์ใหม่ต่างหาก
`forms/main-template-data.js` ที่ดึง base64 จากทั้ง 2 ไฟล์เดิมมาใส่ในตัวแปรคนละชื่อ
(`window.TEMPLATE_1_BASE64`, `window.TEMPLATE_2_BASE64`) โดย**ไม่แตะ**ไฟล์ `template-data.js`
เดิมของแต่ละโมดูลเลย (โมดูลเดี่ยวๆ ยังทำงานเหมือนเดิมทุกประการ)

**กลไกปุ่ม "สร้างไฟล์ทั้งหมด":**
1. อ่านค่า "ชื่อโครงการ" จากช่องกรอกด้านบนของ main
2. โหลด template 1 และ 2 จาก `main-template-data.js` ผ่าน JSZip แยกกัน → แทนที่ `[ชื่อโครงการ]`
   ในแต่ละไฟล์ (ใช้ `replacePlaceholder()` ฟังก์ชันเดียวกับที่ใช้ในทุกโมดูล คัดลอกมาไว้ใน main.html)
3. สร้าง .docx ทั้ง 2 ไฟล์เป็น `Uint8Array` แล้วใส่เข้า **zip ชั้นนอกอีกชั้น** (JSZip ซ้อน JSZip)
4. `generateAsync({type:"blob"})` แล้วดาวน์โหลดเป็น `เอกสารโครงการ_<ชื่อโครงการ>.zip`
   (ข้างในมี `1. ขออนุมัติโครงการก่อสร้าง_<ชื่อ>.docx` และ `2. คำสั่งแต่งตั้งคณะกรรมการ_<ชื่อ>.docx`)

**ทดสอบยืนยันด้วย Node จริง** (จำลอง `window` ด้วย Node's `vm` module เพื่อโหลด
`main-template-data.js` เหมือนเบราว์เซอร์): สร้างทั้ง 2 ไฟล์ผ่าน buildOneDocx() → ตรวจว่า
ชื่อโครงการ (ทดสอบ) ปรากฏถูกต้อง, ไม่มี `[ชื่อโครงการ]` ค้าง, `jc` ของเอกสาร 1 ยังครบ
17 thaiDistribute + 3 center, เอกสาร 2 มีชื่อโครงการซ้ำครบ 3 จุดตามที่ควร → รวม zip ได้ถูกต้อง

**ข้อจำกัดที่ต้องบอกผู้ใช้ (มีคำอธิบายในหน้า main.html ให้แล้ว):** ปุ่ม "สร้างไฟล์ทั้งหมด" ใส่ให้
แค่ **ชื่อโครงการ** เท่านั้น (ฟิลด์เดียวที่ main เก็บอยู่ตอนนี้) ส่วนฟิลด์เฉพาะของแต่ละเอกสาร เช่น
อ้างอิงแผนพัฒนาท้องถิ่น (เอกสาร 1), รายชื่อกรรมการ (เอกสาร 2) **จะยังเป็นวงเล็บเหลี่ยมค้างอยู่ในไฟล์
ที่ได้จากปุ่มนี้** — ถ้าต้องการให้ครบทุกฟิลด์ ต้องเข้าไปกรอก+สร้างจากหน้าของเอกสารนั้นโดยตรงแทน
(หรือรอขยาย main ให้เก็บฟิลด์อื่นเพิ่มในอนาคตตามที่คุยไว้ในรอบที่ 11)

## รอบที่ 13: แก้ข้อมูลหายตอนกลับหน้าแรก + เอาช่องอัปโหลด template ออก (2026-07-17)

ผู้ใช้แจ้ง 2 ปัญหา/คำขอ:
1. กด "กลับไปหน้าแรก" (จากหน้าเอกสารกลับไป main) แล้วชื่อโครงการที่กรอกไว้ที่ main หายไป
   (เพราะกลับไปเป็นการโหลดหน้า main.html ใหม่ทั้งหน้า ไม่ใช่ browser back ที่จำ state เดิมได้)
2. เอาช่องอัปโหลด "ใช้ template.docx ฉบับที่แก้ไขเอง" ออกจากทุกหน้าเอกสาร — ไม่ต้องการใช้แล้ว
   เพราะใช้ template ที่ฝังในไฟล์ (`template-data.js`) อยู่แล้วเสมอ

**แก้ปัญหาข้อมูลหาย:** ทำให้ "ชื่อโครงการ" เดินทางไป-กลับได้ทั้ง 2 ทาง ผ่าน URL query string
(ไม่ใช้ localStorage เหตุผลเดิม — ดูรอบที่ 11):
- main → เอกสาร: ลิงก์เอกสาร 1/2 แนบ `?projectName=...` อยู่แล้ว (ทำไว้ตั้งแต่รอบที่ 11)
- เอกสาร → main (จุดที่ขาดไปก่อนหน้านี้): เพิ่มฟังก์ชัน `updateBackLink()` ในเอกสาร 1 และ 2
  อัปเดต `href` ของลิงก์ "กลับไปหน้าแรก" (`id="backLink"`) ให้แนบ `?projectName=...` ตามค่าล่าสุด
  ในช่องกรอกทุกครั้งที่พิมพ์ (เหมือนกลไกที่ main ใช้อัปเดตลิงก์ไปเอกสาร) แล้วเพิ่ม
  `(function prefillFromDoc(){...})()` ใน `main.html` ให้อ่าน query string ตอนโหลดหน้าแล้วใส่ค่า
  กลับเข้าช่อง "ชื่อโครงการ" — ตอนนี้เดินทางไป-กลับระหว่าง main กับเอกสาร 1/2 กี่รอบ ชื่อโครงการ
  ก็ไม่หายแล้ว (บันทึกข้อความไม่มีฟิลด์ชื่อโครงการ และไม่ได้อยู่ในรายการ main แล้ว เลยไม่ต้องทำ
  กลไกนี้ในหน้านั้น)

**เอาช่องอัปโหลด template ออก:** ลบ `<input type="file" id="templateFile">` และ label กำกับ
ออกจากกล่อง "template-box" ในทั้ง 3 หน้าเอกสาร (`1. ขออนุมัติโครงการก่อสร้าง.html`,
`2. คำสั่งแต่งตั้งคณะกรรมการ.html`, `บันทึกข้อความ.html`) พร้อมลบโค้ด JS ที่เกี่ยวข้องออกทั้งหมด:
`overrideTemplateBuffer` (ตัวแปร + event listener ของ `templateFile`) และเปลี่ยน
`buildDocx()`/`buildOneDocx()` ให้ใช้ `base64ToArrayBuffer(window.TEMPLATE_DOCX_BASE64)` ตรงๆ
เสมอ (เดิมมี `overrideTemplateBuffer ||` นำหน้า) ข้อความอธิบายในกล่อง template-box เปลี่ยนเป็น
บอกแค่ว่าไฟล์ template อยู่ที่ไหน + ถ้าจะแก้เพิ่มให้บอกมาแล้วจะ regenerate ให้ใหม่ (แทนการอัปโหลดเอง)

ตรวจสอบว่าไม่มีโค้ดอ้างถึง `overrideTemplateBuffer`/`templateFile` หลงเหลือ และรัน `node --check`
กับทุก `<script>` ในทั้ง 4 ไฟล์ (3 เอกสาร + main) ผ่านหมด ไม่มี syntax error

## รอบที่ 14: จำข้อมูลทุกฟิลด์ในแต่ละเอกสารด้วย localStorage (2026-07-17)

ผู้ใช้แจ้งว่ารอบที่ 13 แก้ได้แค่ "ชื่อโครงการ" (ฟิลด์เดียวที่เดินทางผ่าน query string ระหว่าง main
กับเอกสาร) แต่ฟิลด์อื่นที่เฉพาะเจาะจงของแต่ละเอกสาร (เช่น **ชื่อกรรมการ** ในเอกสาร 2) ยังหายอยู่
เวลากด "กลับไปหน้าแรก" แล้วย้อนกลับเข้ามาที่เอกสารเดิมใหม่ — เพราะ query string ส่งแค่ projectName
ไม่ได้พาฟิลด์อื่นไปด้วย และ main เองก็ไม่มีที่เก็บฟิลด์เฉพาะเอกสารพวกนี้อยู่แล้ว

**สาเหตุ:** การกด "กลับไปหน้าแรก" คือการโหลดหน้าใหม่ทั้งหน้า (ไม่ใช่ browser cache แบบ back
ปกติ) พอโหลด `main.html` ใหม่ ฟิลด์ที่ main ไม่รู้จัก (ชื่อกรรมการ, อ้างอิงแผนพัฒนาท้องถิ่น ฯลฯ)
ก็ไม่มีที่ไปเก็บไว้เลย พอกลับเข้าเอกสารเดิมอีกครั้งเลยเจอฟอร์มว่างเปล่า

**วิธีแก้:** ใช้ **`localStorage` ของหน้านั้นๆ เอง** (ไม่ใช่ query string ข้ามไฟล์แบบ projectName)
เพราะเป็นการเก็บ-อ่านค่าใน**ไฟล์เดียวกัน**ทุกครั้งที่โหลดซ้ำ ไม่ต้องพึ่ง cross-origin ระหว่างไฟล์
file:// เหมือนตอนที่กังวลไว้ตอนแรก (รอบที่ 11) — ปลอดภัยกว่ามากเพราะเป็น origin เดียวกันแน่นอน
เพิ่มกลไกนี้ในทั้ง 3 หน้าเอกสาร:
- `AUTOSAVE_KEY` เฉพาะของแต่ละหน้า (เช่น `consdocx_autosave_2_คำสั่งแต่งตั้งคณะกรรมการ`)
- `saveAutosave()` — เก็บค่าทุกฟิลด์ของฟอร์มนั้นลง `localStorage` เป็น JSON ทุกครั้งที่มีการพิมพ์
  (ผูกกับ `input` event ของทุกฟิลด์ในฟอร์ม)
- `restoreAutosave()` — อ่านค่ากลับมาใส่ฟอร์มตอนโหลดหน้า (รันก่อน `prefillFromMain()` เสมอ
  เพื่อให้ query string จาก main ที่ "ใหม่กว่า" ทับค่าที่จำไว้ในกรณีฟิลด์ชื่อโครงการโดยเฉพาะได้)
- ครอบ try/catch กัน error กรณี browser ปิด localStorage ไว้ (เช่น private mode บางตัว) ไม่ให้ฟอร์ม
  พังทั้งหน้า

## รอบที่ 15: ปุ่ม "สร้างไฟล์ทั้งหมด" ที่ main ดึงข้อมูลจริงจากแต่ละเอกสารด้วย (2026-07-17)

ผู้ใช้แจ้งว่าไฟล์ .zip ที่สร้างจากปุ่ม "สร้างไฟล์ทั้งหมด" ที่ main มีแค่ชื่อโครงการ (ข้อมูลจาก main
เอง) แต่ฟิลด์ที่กรอกไว้ในแต่ละเอกสารย่อย (เช่น รายชื่อกรรมการในเอกสาร 2, อ้างอิงแผนพัฒนาท้องถิ่นใน
เอกสาร 1) ไม่ติดไปด้วย — ตรงกับข้อจำกัดที่เคยบอกไว้ในรอบที่ 12/14 ว่า main ไม่มีฟิลด์พวกนี้เก็บอยู่

**วิธีแก้ (ใช้ประโยชน์จาก localStorage ต่อฟิลด์ที่ทำไว้ในรอบที่ 14):** ให้ `main.html` เปิดไฟล์เอกสาร
1 และ 2 **ในหน้าที่ซ่อนไว้ (`<iframe style="display:none">`)** พอ iframe โหลดเสร็จ หน้าเอกสารนั้นจะ
รัน `restoreAutosave()` ของตัวเองตามปกติ (อ่านจาก `localStorage` ของตัวเอง — เป็น origin เดียวกับ

กลับมาให้ `main.html` (ใช้ `event.source.postMessage(...)` ตอบกลับตรงไปยัง iframe ที่ถามเท่านั้น)

**สิ่งที่แก้ในโค้ด:**
- `1. ขออนุมัติโครงการก่อสร้าง.html` และ `2. คำสั่งแต่งตั้งคณะกรรมการ.html`: เพิ่ม
  `window.addEventListener("message", ...)` ท้ายสคริปต์ — ถ้าได้รับ `type: "consdocx:getFormData"`

  ชี้ไปที่ path ของเอกสารนั้น รอ `load` event แล้วส่ง postMessage ขอข้อมูล, ฟัง response ที่ตรงกับ
  `docId`, resolve ค่าที่ได้ (หรือ `null` ถ้าเกิน timeout — กันเอกสารพังหรือโหลดไม่ทัน)
- `buildOneDocx()` เปลี่ยนจากรับ `projectName` ตัวเดียวเป็นรับ **`fieldsMap`** (object ของ
  placeholder→ค่า) วนแทนที่ทีละคู่ — รองรับทุกฟิลด์ของแต่ละเอกสารแล้ว ไม่ใช่แค่ชื่อโครงการ
- ปุ่ม "สร้างไฟล์ทั้งหมด" ตอนนี้: (1) ดึงข้อมูลจาก iframe ทั้ง 2 พร้อมกันด้วย `Promise.all` (2) เอา
  ชื่อโครงการจากเอกสารที่ตอบมาก่อน (ถ้าไม่มีเลยค่อย fallback ไปใช้ช่องกรอกที่ main) (3) สร้าง
  เอกสาร 1 ด้วย `[ชื่อโครงการ]` + `[อ้างอิงแผนพัฒนาท้องถิ่น]`, เอกสาร 2 ด้วย `[ชื่อโครงการ]` +
  ชื่อ/ตำแหน่งกรรมการทั้ง 3 คน (4) รวม zip เหมือนเดิม
- ถ้าเอกสารไหนดึงข้อมูลไม่สำเร็จ (timeout/error) จะแจ้งสถานะเตือนแต่ยังสร้างไฟล์ต่อโดยเว้นฟิลด์
  ที่ขาดว่างไว้ ไม่ทำให้ทั้งปุ่มพัง

**ทดสอบยืนยันด้วย Node จริง:** จำลอง `buildOneDocx` แบบใหม่ (`fieldsMap`) ทดสอบใส่ค่าครบทั้ง
ชื่อโครงการ+อ้างอิงแผนในเอกสาร 1 และ ชื่อ/ตำแหน่งกรรมการครบ 3 คนในเอกสาร 2 → ตรวจว่าข้อความ
ปรากฏถูกต้องครบทุกจุด ไม่มี `[...]` หลงเหลือเลยแม้แต่จุดเดียวในทั้ง 2 ไฟล์ และ `node --check`
ผ่านทุกสคริปต์ที่แก้ (หมายเหตุ: ยังไม่ได้ทดสอบ iframe+postMessage จริงในเบราว์เซอร์ เพราะแซนด์บ็อกซ์
นี้ไม่มีเบราว์เซอร์ให้รัน — กลไกนี้ตรวจสอบด้วยการอ่านโค้ดอย่างละเอียด แนะนำให้ผู้ใช้ทดลองกดปุ่มที่

**เงื่อนไขสำคัญที่ต้องบอกผู้ใช้:** ปุ่มนี้จะดึงได้แค่ข้อมูลที่**เคยกรอกและถูกจำไว้แล้ว**

เป็นเหตุผลหลักที่ต้องใช้เทคนิคซับซ้อน (base64 embed หลีกเลี่ยง CORS, iframe+postMessage
หลีกเลี่ยงปัญหา origin ไม่ตรงกันข้ามไฟล์) ถ้าย้ายไปเป็นเว็บไซต์จริงบน domain เดียวกันหมด (Pages)
ปัญหาพวกนี้จะหมดไปเองโดยธรรมชาติ (ทุกหน้าเป็น origin เดียวกันจริงๆ, localStorage/fetch ทำงาน
ตรงไปตรงมา) และผู้ใช้ได้ประโยชน์เพิ่มคือแชร์แค่ลิงก์ ไม่ต้องให้คนอื่นดาวน์โหลดโฟลเดอร์เลย

**สิ่งที่เตรียมไว้แล้วในรอบนี้:** สร้าง `ConsDocx/index.html` ที่ root — เป็นหน้า redirect อัตโนมัติ
ไปที่ `./forms/main.html` (ใช้ทั้ง `<meta http-equiv="refresh">` และ `location.replace()` สำรองกัน
เผื่อ meta refresh ใช้ไม่ได้บาง host) เพราะ GitHub Pages จะ serve `index.html` ที่ root โดย
อัตโนมัติเป็นหน้าแรกเสมอ โครงสร้างโฟลเดอร์ปัจจุบันใช้ path แบบ relative (`./`, `../`) ล้วนอยู่แล้ว
ทุกจุด **ใช้งานได้เลยบน GitHub Pages โดยไม่ต้องแก้โค้ดอะไรเพิ่มอีก** (ไม่ต้องเปลี่ยนจาก base64
embed เป็น fetch เพราะของเดิมก็ยังทำงานได้ปกติบนเว็บจริงเช่นกัน)

**สิ่งที่ผู้ใช้ต้องทำเอง (ไม่มีเครื่องมือ GitHub ให้ผมทำแทนโดยตรงในเซสชันนี้):**
1. สร้าง GitHub repo ใหม่ (public) เช่น `consdocx`
2. อัปโหลด/push เนื้อหาทั้งหมดในโฟลเดอร์ `ConsDocx/` เข้า repo นี้ (รวม `index.html`, `forms/`,
   `templates/`, `shared/` — ไฟล์ `memory.md`/รูปครุฑที่ root จะขึ้นไปด้วยก็ไม่เป็นไร ไม่กระทบอะไร)
3. เข้า repo → Settings → Pages → Source เลือก branch `main` และโฟลเดอร์ `/ (root)` → Save
4. รอสักครู่ จะได้ลิงก์ประมาณ `https://<username>.github.io/<repo>/` — แชร์ลิงก์นี้ให้ใครก็เปิดใช้ได้เลย
   ผ่านเบราว์เซอร์ ไม่ต้องดาวน์โหลดอะไรทั้งสิ้น

**ข้อควรระวังที่บอกผู้ใช้ไปแล้วและผู้ใช้รับทราบ:** repo เป็น public แปลว่าใครก็ดูเนื้อหา template
(รวมชื่อจริงของผู้ลงนาม/ตำแหน่งที่ยังไม่ได้ทำสีแดง) ได้ทั้งหมด — ผู้ใช้ยืนยันว่าโอเค ไม่กังวลเรื่องนี้

## รอบที่ 20: ลอง push GitHub ตรงจากแซนด์บ็อกซ์ไม่สำเร็จ + แนะนำ GitHub Desktop แทน (2026-07-17)

ผู้ใช้ให้ GitHub personal access token (fine-grained) มาเพื่อให้ผม push ไฟล์ขึ้น GitHub repo
ใหม่โดยตรงจากแซนด์บ็อกซ์ (ไม่ผ่านเบราว์เซอร์) — ลองแล้วไม่สำเร็จ: เครือข่ายขาออกของแซนด์บ็อกซ์นี้
ถูกจำกัดด้วย allowlist ผ่าน proxy และ `api.github.com` ไม่ได้อยู่ในรายการที่อนุญาต (ได้
`403 blocked-by-allowlist` จาก proxy ตั้งแต่ขั้นตอน CONNECT ก่อนจะถึง GitHub API ด้วยซ้ำ) ตรวจสอบ
แล้วไม่มี MCP connector ของ GitHub ให้เชื่อมต่อในระบบนี้ด้วย (ค้นด้วย `search_mcp_registry` หลาย
คำค้นแล้วไม่เจอ) และ Claude in Chrome ก็ยังไม่ได้เชื่อมต่อในเซสชันนี้ (extension ไม่ตอบสนอง)

**แนะนำให้ revoke token ที่เคยส่งมาในแชททิ้งทันที** เพราะไม่ได้ใช้งานจริง (การ push ล้มเหลวตั้งแต่
ขั้น network ก่อนจะยิง request ไปหา GitHub เลยด้วยซ้ำ)

**ทางออกระยะยาวที่แนะนำ (ผู้ใช้กังวลว่าจะต้องอัปโหลดเองทุกครั้งที่อัปเดตโปรแกรม):** ใช้
**GitHub Desktop** (แอปฟรี ไม่ต้องพิมพ์คำสั่ง) แทนการอัปโหลดไฟล์ผ่านเว็บทุกครั้ง — เขียนขั้นตอน
setup ครั้งแรกและวิธีอัปเดตในอนาคตไว้ในไฟล์ **`ConsDocx/วิธีอัปเดตโปรแกรมขึ้น GitHub.md`** แล้ว
(สรุปคือ: ติดตั้ง GitHub Desktop → sign in → Add Local Repository ชี้ที่ `D:\ClaudeCowork\ConsDocx`
→ Publish repository เป็น public → หลังจากนั้นแค่กด Commit + Push ทุกครั้งที่มีการอัปเดต) เพราะ
GitHub Desktop เป็นแอปทั่วไป (ไม่ใช่ terminal ที่ถูกจำกัดสิทธิ์แบบ tier "click") ในอนาคตถ้าผู้ใช้
อนุญาตให้ Claude คุมหน้าจอ (computer use) เข้าถึงแอปนี้ได้ Claude จะกดปุ่ม Commit/Push แทนได้เลย
ไม่ต้องให้ผู้ใช้ทำเองทุกรอบ

**สำคัญมาก — ผู้ใช้ย้ำหลายรอบให้ตอบเป็นภาษาไทยเสมอ:** ต้องตอบเป็นภาษาไทยตลอดการสนทนานี้
(อย่าตอบเป็นอังกฤษอีก แม้จะ mix คำอังกฤษเทคนิคได้ตามปกติ) และไฟล์เอกสารสรุป/คำแนะนำที่สร้างให้
ผู้ใช้ (เช่นไฟล์นี้) ก็ควรเป็นภาษาไทยเช่นกัน
